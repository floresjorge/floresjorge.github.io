Simulation ETL & Churn Pipeline

Python pipeline for retail operations data: extract, transform, load, and predict customer churn.

The pipeline processes synthetic sales records from a simulated dog daycare. It validates schema on ingestion, engineers customer-level features, loads to SQLite with idempotent upserts, and trains a logistic regression model with an adjusted classification threshold for the minority churn class.

Three steps from raw CSV to warehouse

The source data is an AppSheet CSV export, the same format produced by the POS system running in production. Three modules handle the work: 

data/raw/sales_export.csv
         │
         ▼
  ┌─────────────┐
  │  extract.py  │  dtype enforcement · schema validation
  └──────┬──────┘  raises ValueError on missing columns or nulls
         │
         ▼
  ┌──────────────┐
  │ transform.py  │  date parsing · service normalization
  └──────┬───────┘  per-customer feature engineering
         │
         ▼
  ┌────────┐
  │ load.py │  SQLAlchemy upsert → SQLite (PK: sale_id)
  └────┬───┘  logs inserted vs updated counts separately
       │
       ├── sales             (10,558 rows)
       └── customer_features (320 rows)

The CLI flag --env dev|prod controls log verbosity and DB path. Dev mode streams DEBUG output to the console; prod mode writes WARNING+ to a log file. Both environments run all validations.

Customer-level feature engineering

The raw data is at the transaction level (one row per sale). transform.py aggregates it to one row per customer for the ML model:

  • days_since_last_visit: days from last visit to 2024-06-30
  • visit_frequency: average visits per month over the customer's active window
  • avg_spend: mean transaction value in MXN
  • preferred_service: modal service, one-hot encoded for the model
  • is_churned: read directly from the raw CSV, never recomputed downstream

The churn label definition lives exclusively in generate_synthetic_data.py as CHURN_CUTOFF_DATE = date(2024, 6, 1). Both transform.py and churn_model.py import this constant instead of hardcoding their own cutoff. One definition, no label drift.

Why threshold 0.35?

The churn class is a 25% minority (80 of 320 customers). At the default threshold of 0.5, a logistic regression tends to under-predict the minority, trading recall for overall accuracy.

Lowering the threshold to 0.35 shifts the trade-off: the model flags more customers as at-risk, accepting a few extra false positives in exchange for catching nearly all true churned customers. For a re-engagement campaign, the cost of a missed churned customer (lost revenue) outweighs the cost of a false positive (one extra email sent).

At threshold 0.35 on the held-out test set: 15 of 16 churned customers correctly identified, 4 active customers flagged as false positives.

Technical decisions

Idempotent upserts

load.py queries existing PKs before inserting, separating records into "new" and "update" batches. This makes the pipeline safe to re-run without creating duplicate rows, which matters when the source CSV is regenerated or the pipeline runs in CI.

Schema-first validation

extract.py defines an EXPECTED_SCHEMA dict at the module level and raises a ValueError on any missing column or null in required fields, before any transformation runs. The pipeline fails fast rather than propagating bad data into the warehouse.

Dev / prod environments

The --env CLI flag switches DB path and logging level simultaneously. No environment variables, no config files. One flag, one change.
